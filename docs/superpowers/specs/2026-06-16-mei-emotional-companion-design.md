# 메이 감정 동반자 + 위기 안전망 — 설계 스펙

- 날짜: 2026-06-16
- 상태: 설계 승인됨 (브레인스토밍 완료, 구현 계획 전)
- 범위: 메이 채팅(`/api/chat`)의 대화 성격 강화 — 위기 안전망 / 회고 지도 / 감정 흐름 인식

## 1. 배경 (현재 상태)

- 메이 채팅은 일기를 근거로 **공감 우선** 응답을 한다(`route.ts` 시스템 프롬프트). 환각 0 정책(일기에 없는 사실 금지)이 강하게 박혀 있다.
- `mood`는 **이미 켜져 있다**: 일기 작성 폼의 `MoodPicker`(😊기쁨/😌평온/😢슬픔/🥰사랑/😤화남/😴피곤)로 선택·저장되고, 채팅 프롬프트에 일기별 `[기분: x]`로 들어간다. (스키마 주석의 "deprecated/베타 미사용"은 stale.)
- 비어 있는 것:
  - **위기 신호에 대한 안전망 전무** (자해·자살 등 → 탐지/자원안내/에스컬레이션 없음).
  - **조언 요청 시 동작 미정의** — 일반 대화 수준으로만 반응.
  - **감정 "흐름"(추세) 인식 없음** — 일기별 기분을 점점이 볼 뿐, 여러 날의 변화는 못 읽음.

## 2. 목표 / 비목표

**목표**
1. 위기 신호를 이중(키워드+모델)으로 감지해 한국 위기상담 자원을 **확정적으로** 안내한다.
2. 감정·조언 대화에서 조언 대신 **본인이 회고하도록 지도**한다(예외적 가벼운 제안만).
3. 이미 쌓이는 mood 데이터로 **감정 흐름**을 읽어 대화 맥락에 맞을 때 자연스럽게 짚는다.

**비목표 (V2 이후)**
- 기분 추세 **차트/시각화 UI**.
- 위험도 판정을 위한 **별도 분류 모델 호출**.
- 임상·의료 서비스화 (메이는 친구이지 상담사/의료진이 아님 — 필요 시 가벼운 고지 문구만).

## 3. 확정된 결정 (브레인스토밍)

| # | 결정 |
|---|---|
| 위기 발동 | **좁게+강하게** — 명확한 자해/자살/학대 신호만. 일상 슬픔은 Phase 2 공감이 담당. |
| 위기 감지 | **인라인 마커 + 키워드 안전망** (둘 중 하나라도 → 발동). 추가 모델 호출 0. |
| 위기 응답 | 차갑게 끊지 않고 공감하며 곁에 있되, **서버가 자원 블록을 상수로 확정 부착**. 위험 감지될 때마다. |
| 회고 지도 | 실용질문=직답 / 감정·조언=회고 우선. **명시적 강요 시에만** 비처방 제안 1~2줄. |
| 감정 흐름 | **관련될 때만 짚어줌** — 대화가 기분 쪽으로 흐를 때만. 데이터 뒷받침될 때만(환각 0 유지). |

## 4. 아키텍처 (한 번의 Gemini 호출, 전·후처리로 감쌈)

```
사용자 메시지
  → [서버 前처리]
       · 키워드 위기 스캔(메시지 텍스트)
       · 감정 흐름 요약 계산 (최근 14일 mood 집계)
  → buildSystemPrompt 에 주입: (안전 지침 + 회고 지침 + 감정흐름 컨텍스트 1줄)
  → chat() — Gemini 1회 호출 (기존과 동일)
  → [서버 後처리]
       · [[risk:high]] 마커 파싱  (기존 extractRefs와 동일 패턴)
       · 위기경로 = (키워드 hit) OR (마커 present)
       · 위기경로면 응답에 crisis 플래그 + 자원블록(상수) 부착
       · [[refs]] 칩 (기존 그대로)
```

- 추가 모델 호출 없음. 기존 마커/스트립 패턴(`[[refs]]`) 재활용.
- 모든 변경은 `src/app/api/chat/route.ts`(프롬프트·후처리), `character-chat.tsx`(위기 블록 렌더), 새 유틸 1~2개(`mood-trend`, `crisis`)에 국한.

## 5. 컴포넌트 B — 위기 안전망 (Phase 1, 단독 출시 가능)

### 5.1 감지 (이중)
- **키워드 스캔(결정적 백업):** 서버에서 사용자 메시지를 정규화 후 위기 키워드 매칭. 시작 목록(큐레이션 필요):
  `죽고싶`, `자살`, `자해`, `죽어버리`, `사라지고 싶`, `살기 싫`, `목을 매`, `뛰어내리`, `없어지고 싶` 등.
  - 재현율 우선(놓침이 오발동보다 나쁨). 오발동은 "혹시 힘들면 여기 있어요" 수준이라 해가 작다.
  - 부정/인용 등 정교한 처리는 하지 않는다(단순 유지). 뉘앙스는 모델 마커가 담당.
- **모델 마커(뉘앙스 층):** 시스템 프롬프트에 *"명확한 자해/자살/학대 위기 신호를 감지하면 따뜻하게 곁에 있는 답을 하고, 맨 끝 줄에 `[[risk: high]]`를 적어라"* 지시. 서버가 파싱 후 본문에서 strip(사용자에겐 안 보임). 파서는 기존 `[[refs]]`와 동일하게 공백·대소문자 허용.
- **위기경로 = 키워드 hit OR 마커 present.**

### 5.2 응답
- 메이는 **차갑게 리다이렉트하지 않는다** — 공감하고 곁에 있는 답을 한다(프롬프트로 유도). 회고/감정흐름 지침보다 안전이 항상 우선.
- 서버가 **위기 자원 블록을 코드 상수로 확정 부착**한다.
  - ⚠️ **전화번호·문구는 절대 LLM이 생성하지 않는다** (틀린 번호 = 최악의 버그). 서버 상수 only.
  - 위험 감지될 때마다 노출(once-only 아님).

### 5.3 자원 블록 (확정 — 2026-06-16 공식 검증)
클라이언트 전용 컴포넌트로 렌더, 번호는 `tel:` 탭 연결.

```
혼자 견디기 힘들 땐 언제든 도움을 받을 수 있어요.
· 자살예방 상담전화 109 (24시간)            tel:109
· 정신건강 위기상담 1577-0199 (24시간)       tel:15770199
```
- 출처: 보건복지부(109 통합 2024.1 시행), 국립정신건강센터(1577-0199 24h/365).

### 5.4 영속화
- 채팅은 영구 보존이므로, 위기 블록도 다시 봐도 보여야 한다.
- `ChatMessage`에 `crisis Boolean @default(false)` 컬럼 추가 → assistant 메시지 저장 시 기록, 로드 시 렌더.
- 마이그레이션 배포 주의: prod `DATABASE_URL` pgbouncer 관련 함정(메모리 `vercel-prisma-pgbouncer-42p05`) 확인.

## 6. 컴포넌트 C — 회고 지도 (Phase 2)

- **분기:** 실용·사실 질문(뭐 했지/어디 갔지/언제)은 **지금처럼 직답**. 감정·고민·조언 요청만 회고 모드.
- **회고 모드 행동:** 공감(일기 근거) → 열린 질문으로 본인이 회고하게(*"그때 어떤 기분이었어?"*, *"비슷한 적 있었어?"*) → **사용자가 명시적으로 강하게 원할 때만** *"참고로…"* 비처방 제안 1~2줄.
- 시스템 프롬프트 `## 응답 스타일`에 회고 지침 추가. 페르소나(따뜻함)·환각 0·구어체 유지.
- 위기경로일 땐 회고보다 안전 응답 우선.

## 7. 컴포넌트 D — 감정 흐름 (Phase 2)

### 7.1 계산 (서버, 결정적 — LLM 추론 아님)
- 새 유틸 `mood-trend`(서버 전용). 입력: userId, now(KST).
- 윈도우: **최근 14일** 일기(`createdAt` 기준)의 `mood` 수집.
- **mood 기록 < 3개 → 흐름 요약 생략**(섹션 자체를 프롬프트에 안 넣음).
- 그 외: 분포 집계 + valence 매핑으로 coarse 라벨 산출.
  - valence: `joy/love/calm` = 긍정, `sad/anger/tired` = 가라앉음(저에너지 포함).
  - ≥60% 가라앉음 → "가라앉은 편" / ≥60% 긍정 → "밝은 편" / 그 외 "기복이 있는 편".
- 출력(프롬프트 주입용 1줄, 근거 사실): 예 `최근 14일 기분기록 6개: 슬픔3·피곤2·평온1 — 가라앉은 편`.

### 7.2 사용 (프롬프트)
- 위 1줄을 새 섹션으로 주입하고 지침: *"이 감정 흐름은 대화가 기분·고민 쪽으로 흐를 때만 자연스럽게 언급해. 뜬금없이 먼저 꺼내지 마. 위 요약을 벗어난 기분 사실은 지어내지 마."*
- 데이터 부족 시 섹션 부재 → 메이는 흐름을 언급하지 않는다.

## 8. 단계 / 순서

- **Phase 1 — 위기 안전망(B):** 안전이 먼저. 감정 토로를 유도하는 C·D 전에 깔린다. 단독 배포 가능. (스키마 마이그레이션 1개 포함)
- **Phase 2 — 감정 동반자(C+D):** 같은 시스템 프롬프트/컨텍스트를 바꾸므로 묶어서.

## 9. 검증 / 성공 기준

- **위기(B):**
  - 키워드 케이스 단위 테스트(대표 위기문장 hit / 일상문장 non-hit).
  - 마커 파싱·strip 단위 테스트(`[[risk:high]]` present/absent, 본문에 노출 안 됨).
  - 위기경로면 자원 블록 항상 부착 + 새로고침 후에도 유지(영속).
  - 번호는 검증된 상수와 정확히 일치(109 / 1577-0199).
- **회고(C):** 샘플 대화 회귀 — 실용질문=직답 / 감정질문=회고. 명시적 조언강요 시 1~2줄 비처방 제안.
- **흐름(D):** `mood-trend` 단위 테스트(빈 데이터 / <3개 생략 / 정상 분포·라벨 경계값). 일기 사실 질문 무회귀.
- **공통:** 기존 환각 0·`[[refs]]` 칩·날짜질문 동작 무회귀.

## 10. 열린 항목 / 리스크

- 위기 키워드 목록은 출시 전 한 번 더 큐레이션(한국어 표현 다양성). 재현율 우선.
- 오발동(false positive) 톤: 자원 블록 문구가 부담스럽지 않게("강요" 아닌 "여기 있어요").
- mood 데이터 희소성: 베타 사용자 mood 기록률이 낮으면 D가 자주 침묵 → 의도된 안전 동작(과장보다 침묵).
- 임상 고지 문구 필요 여부는 Phase 2에서 재검토(현재 비목표).
```
